## PS-4: E-Invoice Format Transcoder

## Description

E-invoices are not stored or exchanged in one universal format. A sender may create an invoice in one format, while the receiver or government network may require a different format.

For example, an invoice from Germany may use XRechnung UBL, while a buyer in France may expect Factur-X, a Malaysian buyer may require PINT UBL, and a Belgian government entity may need Peppol BIS 3.0.

The system should preserve important invoice fields such as seller and buyer details, tax identifiers, issue date, currency, subtotal, tax amount, total amount, payment reference, delivery date, and line items.

This is a data pipeline and validation challenge. The goal is not to generate invoice documents with an LLM. Field mappings, tax rule application, and validation should use deterministic logic.

LLM usage should be minimal. The core solution should use mapping tables, validation rules, feature engineering, deterministic transformations, classical ML, or a hybrid of these.

## Dataset Requirements

You must generate your own synthetic dataset using the schema below.

Required files:

- format_rules.csv: minimum 8 format definition rows
- invoices_source.csv: minimum 3,000 source invoice rows
- mappings_train.csv: minimum 3,000 expected mapping outcome rows
- invoices_test.csv: minimum 800 source invoice rows to transcode and validate

Your generated dataset should include at least 6 distinct source format to target format pairs.

### format_rules.csv

Fields:

- format_id
- format_name
- used_in_countries
- syntax
- required_fields
- optional_fields
- tax_id_field_name
- line_item_structure
- supports_credit_note
- max_line_items

Suggested values:

- format_name: peppol_bis_3, xrechnung_ubl, xrechnung_cii, zugferd, facturx, pint_my, pint_sg, nlcius
- syntax: UBL or CII
- line_item_structure: flat or nested
- required_fields: comma-separated canonical field names

### invoices_source.csv and invoices_test.csv

Fields:

- invoice_id
- source_format
- target_country
- seller_id
- buyer_id
- issue_date
- currency
- subtotal
- tax_rate
- tax_amount
- total_amount
- line_items_json
- seller_vat
- buyer_vat
- payment_reference
- delivery_date

Suggested value ranges:

- subtotal: 50 to 250,000
- tax_rate: 0 to 0.28
- line items per invoice: 1 to 50
- source_format: one of the format_ids in format_rules.csv
- target_country: country that determines the required target format

### mappings_train.csv

Fields:

- invoice_id
- target_format
- is_mapping_valid
- mapping_errors
- corrected_fields

Allowed mapping_errors:

- missing_required_field
- tax_amount_mismatch
- currency_not_supported
- line_item_structure_incompatible
- buyer_vat_missing_for_target
- seller_vat_format_invalid
- credit_note_not_supported

corrected_fields should be a JSON string showing the correct value for each errored field.

Your dataset should include 15-25% invoices with mapping errors.

Final evaluation will use a hidden organizer-generated dataset with the same schema. Your solution should generalize beyond your own generated data.
